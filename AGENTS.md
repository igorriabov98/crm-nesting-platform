## Запрещено без явного подтверждения оператора

- `git push` в любой remote.
- Деплой в любую среду, включая Vercel, Railway, Supabase и другие production/staging/preview окружения.
- `prisma migrate deploy`, `prisma migrate reset` или любые команды миграций, которые меняют БД.
- Изменение production environment variables, secrets, доменов, build/deploy settings или runtime settings.
- Удаление данных, storage objects, database rows, buckets, deployments, branches или remote resources.

Перед любым таким действием нужно отдельно получить явное подтверждение оператора в текущем диалоге.

## Правила параллельной работы и деплоя

1. ОДИН ПИСАТЕЛЬ В MAIN. `main` меняется только merge'ем, который выполняет один процесс. Перед merge: `git fetch --all --prune` и проверка, что локальный `main` == `origin/main`. Force-push в `main` запрещён всегда.
2. АГЕНТ = СВОЯ ВЕТКА = СВОЙ WORKTREE. В чужие worktree и ветки не писать. Оператор правит руками - в отдельной ветке.
3. ДЕПЛОЙ ТОЛЬКО ИЗ MAIN, только после push. Railway и Vercel должны деплоиться с одного SHA. Деплой с feature-веток запрещён.
4. ПЕРЕД ЛЮБОЙ ЗАДАЧЕЙ: `git fetch --all --prune` и отчёт первой строкой: `база: origin/main <sha>`. Если ветка устарела от `main`, сначала merge `main` в ветку, потом работа.
5. ПОСЛЕ MERGE В MAIN: `git log origin/main --oneline -3` в отчёт как доказательство, что коммиты реально в `origin/main`.
6. Секция "Запрещено без явного подтверждения оператора" действует поверх всего: push, deploy, миграции, изменение env, удаление данных.
7. Каждый отчёт агента заканчивается блоком: ветка, HEAD SHA, что в origin (запушено/нет), что в проде (задеплоено/нет).

## Защита main (нет technical enforcement)

Branch protection для приватного репо может быть недоступен или не включён. GitHub технически может не заблокировать прямой push в `main`, force-push или merge с красным CI. В таком режиме защитой является этот файл и дисциплина агента.

1. ПРЯМОЙ PUSH В `main` ЗАПРЕЩЁН. Всегда использовать путь: отдельная ветка -> PR -> зелёный CI -> merge. Это действует даже для однострочных правок, документации и срочных исправлений.
2. FORCE-PUSH В `main` ЗАПРЕЩЁН НИКОГДА. История `main` неприкосновенна.
3. MERGE РАЗРЕШЁН ТОЛЬКО ПРИ ПОЛНОСТЬЮ ЗЕЛЁНОМ CI, включая job `real fixtures` (боевые STEP: LEDA.024, LEDA.525). Красный `real fixtures` означает регресс на реальном изделии; merge запрещён, даже если остальные job зелёные. Если GitHub не блокирует такой merge технически, агент блокирует его сам.
4. Перед merge агент ОБЯЗАН явно перечислить в отчёте статусы всех четырёх job: `checks`, `root build`, `tests`, `real fixtures`. Формулировка "CI зелёный" без перечисления job недостаточна.
5. Любое нарушение этих правил возможно только по явному письменному подтверждению оператора в текущем диалоге. Формулировку исключения пишет сам оператор.

## Инфраструктурные грабли (известные)

- Новый worktree не содержит `node_modules`; перед `build` или тестами выполнить установку зависимостей тем менеджером, который задан lock-файлом (`npm ci` для `package-lock.json`).
- Полная `next build` требует env (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` и другие). Локально без `.env.local` сборка может падать на пререндере страниц. Для проверки использовать CI-dummy-env или ограничиться `tsc --noEmit`, если задача не касается full build.
- CAD-фикстуры (`*.STEP`, `*.step`, `*.stp`) могут содержать trailing whitespace из CAD-экспорта. Это не ошибка геометрии. Отключать whitespace-проверку через `.gitattributes` (`-text -whitespace`), геометрию руками не править.
- `tsx --eval` компилирует сниппеты как CJS и не поддерживает top-level `await`; использовать async IIFE или временный файл-скрипт.

## UI-правило для выпадающих списков

Во всех выпадающих меню закрытое состояние должно показывать тот же человекочитаемый label, что и элемент в открытом списке. Нельзя оставлять технические значения enum/id вроде `standard`, `undefined` или UUID; если используется `SelectValue`, передавайте отображаемый текст явно.

## Уроки цикла ЛЕДА.525 (эмпирические правила)

1. ИНВАРИАНТ СВЕРЯЕТ НАЙДЕННОЕ С ИСПОЛЬЗОВАННЫМ, не с собой. Примеры: `totalBodies` сверяется с числом solid-экземпляров STEP, `expectedForTree` не выводится из уже подключённых полок, `found==connected` сравнивает найденные bend-пары с реально подключёнными. Инвариант, чей эталон выведен из проверяемого результата, тавтологичен.
2. ТЕСТ ПРОВЕРЯЕТ СВОЙСТВО, НЕ СУРРОГАТ. Число вершин вместо площади, `points > N` вместо наличия дуги/язычка, bbox-высота вместо площади контура - суррогаты. Проверяйте свойство напрямую: точку дуги, признак язычка, площадь shoelace, конкретную связь тел.
3. У ГЕОМЕТРИИ КОНТУРА НУЖЕН ИНВАРИАНТ ФОРМЫ, НЕ ТОЛЬКО МЕРЫ. Площадь, габарит и число вершин могут совпасть на плохой форме. Форма развёртки проверяется как набор: простой внешний контур, площадь итогового cut-контура против `volume/thickness`, вершины внутри bbox заготовки, отсутствие наложений компонентов, сохранённые дуги и ожидаемые признаки фигурности.
4. ФИЗИКА, НЕ ЭВРИСТИКА для геометрических решений. Знак, сторона и ось гиба берутся из нормалей, линии сгиба и соответствия концов общего ребра. `boundsArea`, `score` и похожие эвристики допустимы только как tie-breaker после физических ограничений, а не как источник истины.
5. БОЕВОЙ STEP ГРЯЗНЕЕ СИНТЕТИКИ. Реальные детали могут дать косые гибы, фрагментацию цилиндров, вырезы в зоне гиба и смешанные ветви, которых нет в чистых двойниках. Синтетика полезна для регресса после того, как боевая деталь вскрыла баг.
6. ИСТИНА - В ФАЙЛЕ НА ПЛАЗМУ, не в метрике или локальном артефакте. Reconciliation и preview могут быть зелёными, но рез проверяется по DXF/SVG, которые реально уходят оператору и плазме. Для спорных форм открывайте итоговый DXF/cut-контур.
