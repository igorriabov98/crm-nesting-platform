# Инструкция по запуску `nesting-service`

Проект находится здесь:

```text
\\Mac\Home\Desktop\Tehnolog\nesting-service
```

Сервис больше не использует BullMQ, Redis и Docker. Очереди работают через `pg-boss`, а `pg-boss` хранит свои таблицы в PostgreSQL в отдельной schema `pgboss`. Эта schema создаётся автоматически при первом запуске API или worker-процесса и не конфликтует со schema `nesting`.

## 1. Главные файлы

```text
supabase/nesting_manual_setup.sql  - ручное создание schema nesting и seed-данных
.env.example                       - пример настроек
.env                               - локальные настройки сервиса
src/server.ts                      - Fastify API
src/lib/queue.ts                   - pg-boss singleton и имена очередей
src/workers/step-worker.ts         - worker парсинга STEP
src/workers/nesting-worker.ts      - worker расчёта раскладки
prisma/schema.prisma               - Prisma schema
ecosystem.config.js                - PM2 конфиг на 3 процесса
```

Файла `docker-compose.yml` больше нет, потому что Redis больше не нужен.

## 2. PostgreSQL и ручной SQL

Если структура `nesting` ещё не создана в Supabase:

1. Открой Supabase project.
2. Перейди в `SQL Editor`.
3. Создай новый query.
4. Открой файл:

```text
\\Mac\Home\Desktop\Tehnolog\nesting-service\supabase\nesting_manual_setup.sql
```

5. Выполни весь SQL.

Этот SQL создаёт:

```text
nesting."NestingProject"
nesting."Part"
nesting."NestingSheet"
nesting."SheetCatalog"
nesting."GapTable"
nesting."KFactor"
nesting."Remnant"
```

`pgboss` вручную создавать не нужно. `pg-boss` сделает это сам.

## 3. `.env`

В `.env` должен быть рабочий PostgreSQL URL:

```env
DATABASE_URL=postgresql://USER:PASSWORD@HOST:5432/postgres?sslmode=require&schema=nesting
```

Если в URL уже есть параметры, добавляй `schema=nesting` через `&`. Если параметров нет, добавляй через `?`.

Redis-переменных больше нет:

```text
REDIS_HOST
REDIS_PORT
REDIS_PASSWORD
```

## 4. Установка зависимостей

На Windows используй `npm.cmd`, потому что PowerShell может блокировать `npm.ps1`.

Для UNC-пути лучше запускать через `pushd`:

```powershell
cmd /d /c 'pushd "\\Mac\Home\Desktop\Tehnolog\nesting-service" && npm.cmd install && popd'
```


```json

  "status": "ok",
  "services": {
    "database": { "status": "ok", "latencyMs": 5 },
    "queues": {
      "stepParsing": { "queued": 0 },
```

```json
  "status": "down"

```

Статуса `degraded` больше нет, потому что Redis больше не проверяется.

## 9. Workers

STEP worker:

```powershell
cmd /d /c 'pushd "\\Mac\Home\Desktop\Tehnolog\nesting-service" && npm.cmd run worker:step && popd'
```

Nesting worker:

```powershell
cmd /d /c 'pushd "\\Mac\Home\Desktop\Tehnolog\nesting-service" && npm.cmd run worker:nesting && popd'
```

STEP worker обрабатывает одну задачу за раз. Nesting worker регистрирует два обработчика pg-boss в одном процессе, поэтому может обрабатывать две задачи параллельно.

## 10. Проверка очереди

1. Запусти API.
2. Запусти `worker:step`.
3. Создай проект:

```powershell
curl.exe -X POST http://localhost:4000/api/projects ^
  -F "stepFile=@test.step" ^
  -F "orderNumber=PGBOSS-TEST"
```

Ожидание: проект перейдёт в статус `parsing`, а STEP worker напишет в лог, что получил job.

Проверить schema pg-boss в PostgreSQL:

```sql
SELECT schema_name
FROM information_schema.schemata
WHERE schema_name = 'pgboss';
```

## 11. PM2

```powershell
cmd /d /c 'pushd "\\Mac\Home\Desktop\Tehnolog\nesting-service" && npm.cmd run build && npx.cmd pm2 start ecosystem.config.js && npx.cmd pm2 status && popd'
```

Ожидается 3 процесса:

```text
nesting-api
step-worker
nesting-worker
```

API запускается в `fork` mode и в одном instance. PM2 cluster mode для этого сервиса не используется.

Остановить PM2-процессы:

```powershell
cmd /d /c 'pushd "\\Mac\Home\Desktop\Tehnolog\nesting-service" && npx.cmd pm2 stop all && npx.cmd pm2 delete all && popd'
```
