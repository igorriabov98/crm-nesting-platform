# Правила проекта (НЕЛЬЗЯ нарушать)

Этот документ содержит строгие технические и бизнес-аксиомы CRM-системы. Любое отступление от этих правил ломает логику продукта.

## 🏢 Бизнес-правила

### Изоляция Заводов
- [ ] Данные заводов Бергово и Ужгород полностью изолированы.
- [ ] Каждый пользователь системы привязан к ОДНОМУ `factory_id`.
- [ ] Данные (пользователи, машины, этапы, инвойсы, номенклатуры) показываются ТОЛЬКО для завода сотрудника.

### Распределение Ролей
- [ ] Создание новых пользователей и блокировка старых — ТОЛЬКО роль `planning_director`.
- [ ] Финансы и инвойсы (таблица `invoices`) видят ТОЛЬКО: `planning_director`, `financial_director`, `commercial_director`, `sales_manager`.
- [ ] Изменять статусы инвойсов на `paid` имеют право ТОЛЬКО: `financial_director`, `sales_manager`, `planning_director`.
- [ ] Строгое колоночное ограничение `supply_items`: каждая должность по снабжению может редактировать ТОЛЬКО свой кластер колонок (Инженер — только флаг подтверждения; Технолог — номенклатуру, кол-во; Снабжение — цены и поставщиков). Обеспечено триггером `check_supply_items_column_update()`.

### Логика Производства (Machines & Stages)
- [ ] Заготовка (`cutting`) — ВСЕГДА в Цех 1 (`workshop = 1`). Смена запрещена.
- [ ] Малярка (`painting`) — ВСЕГДА Цех 2.
- [ ] Упаковка (`packaging`) — ВСЕГДА Цех 2.
- [ ] Сборка, Зачистка, Цинк, Отгрузка — выбор Цех 1 или 2, либо отсутствие цеха (для отгрузки).
- [ ] Если покрытие контракта = "цинк" (`coating` = `zinc`) → этап Цинкования нельзя пропустить (`is_skipped` жестко заблокирован в ложь). В остальных случаях этапы пропустить можно.
- [ ] Хронология: `date_end` всегда должно быть больше или равно `>=` `date_start`.

### Математика Снабжения
- [ ] Дедлайн Технолога = `planned_delivery_date` минус 10 дней.
- [ ] Дедлайн Инженера = Дедлайн Технолога минус 2 дня.
- [ ] Автопересчёт: триггер мгновенно меняет эти значения при любых изменениях даты плановой поставки.

### Логика Инвойсов
- [ ] Инвойс создаётся АВТОМАТИЧЕСКИ (UPSERT) при установке даты `date_end` в этапе `shipping`. НЕЛЬЗЯ создавать инвойс вручную через интерфейс.
- [ ] Расчет лимита: Дата оплаты (`payment_date`) = Фактическая отгрузка + 14 дней.
- [ ] Устаревание: Переход в статус `overdue` происходит автоматически через ежедневную фоновую микро-проверку (CRON).

### Алгоритм Уведомлений
- [ ] Уведомления поступают ТОЛЬКО во внутреннюю CRM систему (в "колокольчик" через дашборд), использование сторонних сервисов (email, SMS, Telegram) не предусмотрено.
- [ ] Нет дублей: Перед отправкой CRON проверяет, не отправлено ли аналогичное уведомление в рамках `CURRENT_DATE`.
- [ ] Пользователь получает и видит оповещения строго направленные на его UUID (`user_id = auth.uid()`).

## 💻 Технические правила разработки

- [ ] **TypeScript STRICT:** НИКОГДА не применять заглушку `any`. Код должен иметь полные интерфейсы.
- [ ] **Data Security:** Функционал RLS (`Row Level Security`) обязан быть активирован на ВСЕХ таблицах БД. Любые SELECT и UPDATE должны быть защищены политикой `authenticated`.
- [ ] **Factory Firewall:** Все запросы выборки (Select Policies) защищены WHERE `factory_id = get_user_factory_id()`.
- [ ] **Сидеть на UUID:** Использовать UUIDv4 встроенного типа Postgres `gen_random_uuid()` для всех `id`.
- [ ] **Таймзона:** Все даты (`now()`, `timestamptz`) хранятся и обрабатываются строго по стандарту UTC.
- [ ] **Cascade:** Обязательное CASCADE DELETE: при удалении машины, все её производственные этапы `production_stages`, позиции снабжения `supply_items`, уведомления `notifications` и `invoices` уничтожаются сервером.
- [ ] **Next.js Paradigms:** Клиентские компоненты (`"use client"`) используются умеренно, только при работе с хуками. По умолчанию пишутся Server Components.
- [ ] **No Hardcodes:** Переменные ролей, статусов, типов покрытий импортируются исключительно из централизованных `constants`.
- [ ] **No Custom Backend:** Серверные действия реализуются через интеграции клиента Supabase (`@supabase/supabase-js`) и Server Actions. Кастомные Express/Node хэндлеры через API routes `/api/*` не делать.
## Обновление 27.04.2026: обязательные бизнес-правила Parts 1-3
- Машина создается без завода. `factory_id` нельзя требовать в форме создания.
- Завод назначается на собрании или вручную директором через AssignFactoryDialog.
- Только директорские роли могут назначать завод и управлять собраниями.
- `production_manager` видит только свой завод и не управляет глобальным заводским фильтром.
- Машина без завода не показывается в Production и Gantt.
- Машина без завода показывается в Sales Plan, Supply и Dashboard как отдельная зона внимания.
- Покрытие хранится только в `machine_items.coating`; обращаться к `machines.coating` запрещено.
- Gantt ведется отдельно по заводам Бергово и Ужгород.
- Собрания бывают общие и заводские; решения собрания могут назначить завод и тип материала машине.
