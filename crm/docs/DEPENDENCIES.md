# Карта зависимостей (Dependencies)

В этом документе описаны архитектурные и логические связи между сущностями CRM.

## 1. Зависимости создания (Кто от чего зависит)
Иерархия создания записей в системе имеет древовидную структуру (CASCADE DEPENDENCY):
```text
factories (seed, создаётся 1 раз при развертывании)
  └── users (каждый пользователь привязан к конкретному заводу)
        └── machines (создаёт sales_manager)
              ├── production_stages (создаются АВТОМАТИЧЕСКИ базой при создании машины)
              ├── supply_items (создаются вручную технологом/снабжением)
              └── invoices (создаётся АВТОМАТИЧЕСКИ сервером при отгрузке машины)
                    └── notifications (генерируются триггерами/событиями)
```

## 2. Зависимости данных (Откуда что берётся)

### machines → production_stages
- **Размножение:** При создании 1 машины, триггер БД создает 7 заготовок этапов производства (`cutting`, `assembly`, `cleaning`, `galvanizing`, `painting`, `packaging`, `shipping`).
- **Правило цинкования:** Если у машины `coating = 'zinc'`, этап `galvanizing` защищается от пропуска (`is_skipped` нельзя выставить в `true`). При `coating ≠ 'zinc'`, этот этап может быть пропущен.
- **Автоматическое назначение цехов (workshop):**
  - `cutting` → 1 (жесткая привязка)
  - `assembly`, `cleaning`, `galvanizing`, `shipping` → NULL (оставляет выбор за начальником производства)
  - `painting`, `packaging` → 2 (жесткая привязка)

### machines → supply_items
- Отношение 1 ко многим. Одна машина обслуживается любым количеством позиций снабжения.
- Данные (id машины) подтягиваются автоматически. Доступ к чертежам и тоннажу осуществляется исключительно через SQL `JOIN` к таблице `machines` во избежание денормализации БД.

### machines → invoices
- Отношение 1 к 1 (`UNIQUE machine_id` в таблице `invoices`).
- Сумма чека (`invoice.amount`) копируется из `machines.invoice_amount`.
- Инвойс генерируется автоматически только когда завершается производственный цикл (отгрузка).

### supply_items → автоматические дедлайны
- `technologist_deadline` = `planned_delivery_date` МИНУС 10 дней.
- `engineer_deadline` = `technologist_deadline` МИНУС 2 дня.
- Перерасчет этих значений происходит триггером "на лету" при любом `UPDATE` связанного с `planned_delivery_date`.

### production_stages → invoices
- При обновлении (UPDATE) этапа `shipping`, когда `date_end` заполняется:
  - Выполняется операция `UPSERT`:
    - Если инвойса еще нет → выполняется `CREATE` (со статусом `not_paid`).
    - Если инвойс уже есть → выполняется корректировка `payment_date`.
- Жесткая связь значения: `payment_date` = `shipping.date_end` + 14 дней.

### invoices → автоматический статус
- **Устаревание:** если текущая дата (today) превышает `payment_date` и счет всё еще в статусе `'not_paid'`, статус автоматически меняется на `'overdue'`. Реализуется через фоновое задание (Cron).

## 3. Зависимости удаления (CASCADE)
Используется каскадное удаление данных средствами PostgreSQL для предотвращения "сиротских" записей:
```text
Удаление машины (DELETE from machines) приводит к транзакционному каскаду:
  → Удаляются все привязанные production_stages
  → Удаляются все supply_items
  → Удаляется invoice
  → Удаляются все notifications для данной машины
```

## 4. Зависимости валидации и бизнес-правила
На уровне базы данных (триггеров) жестко зафиксировано:
1. Невозможно пропустить (skip) этап `galvanizing`, если покрытие (`coating`) = `'zinc'`.
2. `cutting.workshop` всегда равен 1. Попытка изменить вызывает исключение `RAISE EXCEPTION`.
3. `painting.workshop` всегда равен 2.
4. `packaging.workshop` всегда равен 2.
5. Защита времени: `date_end` не может быть исторически раньше `date_start`.
6. Значение `tonnage` должно быть > 0.
7. Значение `invoice_amount` >= 0.
8. `quantity` > 0, `price_per_unit` >= 0 (защита целостности).
